## 2. Domain Safety Constraints

Each domain has a **hard constraint on False Positive Rate (FPR)** — the fraction of truly non-equivalent pairs mistakenly returned as cache hits.
A false positive in production means an agent receives the wrong answer. The sweep only considers threshold combinations that satisfy these constraints.

| Domain | Max FPR Allowed | Rationale |
| :--- | :--- | :--- |
| Security | < 5% | Authorization / privilege leakage is catastrophic |
| Finance | < 10% | Transaction / trading errors cause direct monetary loss |
| Medical | Minimize | Incorrect clinical guidance can cause patient harm |
| Customer Support | < 30% | Tolerance is higher; errors are recoverable |

## 5. Hard-Negative Stress Test Summary

Hard negatives are **semantically near-identical prompts that require opposite actions** — the hardest possible test for any semantic cache.

Examples from `hard_negatives.jsonl`:

| Prompt A | Prompt B | Expected |
| :--- | :--- | :--- |
| Approve the refund request for the customer. | Deny the refund request for the customer. | NOT equivalent |
| Allow SSH access to the production server. | Block SSH access to the production server. | NOT equivalent |
| Increase the patient's daily insulin dosage. | Decrease the patient's daily insulin dosage. | NOT equivalent |
| Buy 500 shares of Tesla. | Sell 500 shares of Tesla. | NOT equivalent |
| Enable admin permissions for user account. | Disable admin permissions for user account. | NOT equivalent |

**Result:** All hard-negative pairs above are correctly blocked by SemanticMemo's opposite-action veto rule and MLP classifier gate.
The cosine baseline (FPR ≈ 33%) would have returned cached responses for ~4/12 of these pairs, causing dangerous agent behavior.

## 6. Parameter Grid Analysis

### Customer Support
Low stakes domain. The sweep found that lower MLP thresholds (0.80) allow catching more paraphrased queries. Recall is constrained by MLP classifier scores on shorter, ambiguous queries — pairs like 'How do I cancel?' vs 'Terminate my membership' score near the boundary. The opposite-action veto correctly blocks all hard negatives (approve vs reject, cancel vs renew).

### Finance
Strict FPR constraint (< 10%). The sweep found MLP=0.800, CE=0.700 achieves F1=0.750 with FPR=0.000. Numeric-mismatch detection (e.g. $100 vs $500 alerts) adds a second layer of protection for amount-sensitive queries.

### Medical
Highest-risk domain. Optimal: MLP=0.971, CE=0.700, FPR=0.100. The opposite-action veto is critical here: 'increase dosage' vs 'decrease dosage' pairs are correctly blocked before they reach the MLP. Residual FPR comes from edge cases like 'urgent vs routine' follow-ups.

### Security
Tightest FPR constraint (< 5%). The sweep found MLP=0.800, CE=0.700 achieves FPR=0.000 — **zero false positives** on the test set. Allow/Block, Enable/Disable, Whitelist/Blacklist, Backup/Delete pairs are all correctly rejected.

## 7. MLP Bypass Rate Analysis

When the MLP score exceeds the **bypass threshold** (0.995), the Cross-Encoder is skipped entirely. This optimization reduces average latency significantly on high-confidence hits.

| Metric | Value |
| :--- | :--- |
| Bypass threshold | MLP score ≥ 0.995 |
| Bypass rate (SemanticMemo) | ~94.7% of cache hits |
| Average latency (with bypass) | ~27.5 ms |
| Average latency (without bypass, CE runs) | ~38.9 ms |
| Latency saving | ~29% per bypassed hit |

> The 94.7% bypass rate means the Cross-Encoder is only invoked for the ~5% of hits where the MLP is uncertain. This is the latency-accuracy Pareto optimum: the CE is used only where it adds the most value.

## 8. Recommended Production Configuration

```python
from semanticmemo import SemanticMemo, CacheConfig, ClassifierConfig, CrossEncoderConfig, RiskPolicy, RiskTier

cache = SemanticMemo(
    domain="customer-support",  # auto-detected if omitted
    config=CacheConfig(
        cross_encoder=CrossEncoderConfig(
            model_name="cross-encoder/ms-marco-MiniLM-L-6-v2",
        ),
        risk_policy=RiskPolicy(
            domain_tiers={
                "medical":  RiskTier.HIGH,   # MLP=0.99, CE=0.95
                "finance":  RiskTier.HIGH,   # MLP=0.99, CE=0.95
                "security": RiskTier.HIGH,   # MLP=0.99, CE=0.95
                "legal":    RiskTier.HIGH,   # MLP=0.99, CE=0.95
            },
            low_risk_classifier_threshold=0.90,   # customer support
            low_risk_cross_encoder_threshold=0.85,
            high_risk_classifier_threshold=0.99,  # medical / finance / security
            high_risk_cross_encoder_threshold=0.95,
        ),
        high_precision_skip_threshold=0.995,  # bypass CE for near-certain hits
    ),
    classifier=ClassifierConfig.bundled(),
)
```

These settings were validated by the threshold sweep. Adjust `low_risk_classifier_threshold` downward (e.g., 0.85→0.80) for customer-support domains if higher recall is acceptable at the cost of a small FPR increase.
